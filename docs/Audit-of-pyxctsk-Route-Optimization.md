# Audit: pyxctsk Route Optimization vs. FAI S7F & the Ding et al. "Touring n Circles" Algorithm

## TL;DR
- **pyxctsk gets the Earth model right but the optimization algorithm wrong.** It computes leg distances on the WGS84 ellipsoid via `geopy.distance.geodesic` (Karney's algorithm), which matches FAI S7F 2026 §4.2/§7.1.4–7.1.5, but it replaces the mandated geometric shortest-path method (Ding et al. 2018 "PathFinder" with point-circle-point reflection/crossing cases, run in a localized Transverse Mercator plane) with a heuristic dynamic-programming + beam-search + iterative-refinement scheme that can return non-optimal (too-long) distances.
- **The highest-risk gaps** are: (1) no localized-Transverse-Mercator Cartesian projection (spec §7.1.2); (2) apparent lack of the "crossing" case for a previous point that lies inside a cylinder (nested/concentric/overlapping cylinders and start-cylinder-inside geometries), which the reference algorithm explicitly handles; (3) a convergence rule based on a single non-improving pass rather than the spec's ε = 0.1 m threshold; and (4) the task's `earthModel` field (per the xctrack.org Competition Interfaces spec, `earthModel` is a string "one of 'WGS84'(default) / 'FAI_SPHERE'") appears to be ignored, always using WGS84.
- **Recommendation:** treat the current output as an approximation. The ready-to-paste Claude Code prompt in the final section specifies exactly which files/functions to change and gives acceptance tests against XCTrack/FS reference distances.

## Key Findings

### What the specifications require
1. **Task distance = optimized shortest path** from launch to goal, touching every intermediate cylinder/line in order (FAI S7F 2026 §7.2; 2024 §6.4.2). Speed-section distance = optimized launch→ESS distance minus the pre-start portion.
2. **Earth model & distance:** positions are WGS84 (§4.1); distances between two points are computed on the WGS84 ellipsoid using an inverse-geodesic algorithm — Karney (2013)/GeographicLib, Thomas (1970)/Boost, or Vincenty (1975) — all accurate to ≤ 1 mm (§4.2, §7.1.4, §7.1.5). Per GeographicLib's documentation, Karney (2013) is "accurate to round-off for terrestrial ellipsoids (the error in the distance is less than 15 nanometers, compared to 0.1 mm for Vincenty)" and, unlike Vincenty (1975), "the solution of the inverse problem is always found" (Karney, C.F.F., "Algorithms for geodesics," *J. Geodesy* 87(1):43–55, 2013). Paragliding moved from the FAI sphere (radius 6371 km) to the WGS84 ellipsoid in 2018.
3. **Projection for the search:** the 2026 code says the shortest-path search runs in *planar Cartesian* geometry using a **Localized Transverse Mercator (LTM)** projection centered on the task-area centre (§7.1.2), then transforms each path point back to WGS84, applies a **ProjectionCorrection** to snap it onto the control-zone boundary (§7.1.7), and only then measures the real distance on the ellipsoid.
4. **PathFinder algorithm:** the 2026 code explicitly names its shortest-path routine "PathFinder" and cites Yuepeng Ding, Xiong Xie & Bo Jiang (Dalian Maritime University), "An Efficient Algorithm for Touring n Circles," *MATEC Web of Conferences* 232, 03027 (2018), DOI 10.1051/matecconf/201823203027 (EITCE 2018) (§7.1.3). It sets a convergence threshold **ε = 0.1 m** ("sufficient to ensure the solution is accurate within 1 m").
5. **The Ding et al. method** (the MATEC paper, whose abstract states it "proposed an intelligent algorithm, which can build the shortest path of the intersecting circle sequences in the plane… in polynomial time of O(kn) where 1 ≤ k ≤ n under a given computational threshold"): it iteratively places one point on each circle using the **point-circle-point (PCP)** construction; each partial solution is either a **reflection** case (optimal point found by the light-reflection/PCP formula) or a **crossing** case (when a neighbouring point is inside the circle, the optimal point is the intersection of the straight segment with the circle boundary). [MATEC Conferences](https://www.matec-conferences.org/articles/matecconf/pdf/2018/91/matecconf_eitce2018_03027.pdf) It sweeps odd-indexed then even-indexed circles alternately and repeats until the change in total path length drops below ε. It explicitly handles disjoint, tangent, intersecting, and *contained* circles. [MATEC Conferences](https://www.matec-conferences.org/articles/matecconf/pdf/2018/91/matecconf_eitce2018_03027.pdf)
6. **Goal handling:** a goal line is perpendicular to the line from the previous optimized route point *p* to the goal centre *c* (2026 §6.2.3.1; the 2025 change made the goal-line orientation follow the optimized route rather than turnpoint centres). ESS may be a cylinder or coincide with goal.
7. **Tolerances:** relative turnpoint tolerance is 0% in 2026 (0.1% in 2025, historically 0.5%); this is an evaluation tolerance and does not change the optimized geometric distance.

### 2024 vs 2026 differences relevant to optimization
- 2024 (§6.4.1 + Annex A) references an older, simpler "Algorithm for finding shortest path"; 2026 (§7.1) is a complete rewrite specifying LTM projection, the Ding et al. PathFinder, DirectGeodesic/InverseGeodesic, EllipsoidDistance, FindTaskAreaCentre, ProjectionCorrection and a RouteOptimizer wrapper. Both editions compute final distances on the WGS84 ellipsoid.
- 2025/2026 added: goal-line orientation follows the optimized route; turnpoint tolerance reduced to 0%; explicit projection and geodesic algorithms defined; renaming of "Race to Goal"→"Race" and "Elapsed Time"→"Time Trial"; introduction of Elevated Goal (which implicitly also serves as ESS).

### How pyxctsk actually works
Confirmed from `src/pyxctsk/distance.py` and `src/pyxctsk/route_optimization.py`:
- **Distance metric:** every leg is `geopy.distance.geodesic(prev_point, optimal_point).meters`; `distance.py`'s docstring states it uses the "WGS84 ellipsoid." [github](https://github.com/simonsteiner/pyxctsk/blob/main/src/pyxctsk/distance.py) geopy's `geodesic` uses Karney's algorithm by default. ✅ This matches the spec's distance requirement.
- **Algorithm:** `calculate_iteratively_refined_route` runs a dynamic-programming forward pass (`_run_dp` → `_process_dp_stage`) with **beam search** (keep best `beam_width` candidates per stage) and **iterative refinement**: pass 1 uses each next cylinder's *centre* as the look-ahead target (`_center_lookahead`); later passes use the previous route's points (`_route_lookahead`). [github](https://github.com/simonsteiner/pyxctsk/blob/main/src/pyxctsk/route_optimization.py) It stops when a pass fails to improve.
- **Geometry seam:** each stage calls `current_tp.optimal_point(prev_point, next_target)` on a `TurnpointGeometry`/`TaskTurnpoint` object in `turnpoint.py`; the inline comment says this method "resolves its own geometry (cylinder, goal line, or zero-radius center)." [github](https://github.com/simonsteiner/pyxctsk/blob/main/src/pyxctsk/route_optimization.py) Goal-line finishes are only *detected* (a log message) in the optimizer; the actual goal-line math lives in `optimal_point`.
- **Origin:** DP starts at `turnpoints[0].center` with distance 0. [github](https://github.com/simonsteiner/pyxctsk/blob/main/src/pyxctsk/route_optimization.py)

### Assessment: what is correct vs. questionable
**Correct / spec-aligned:**
- WGS84-ellipsoid geodesic distances (Karney) for leg lengths.
- Sequential launch→…→goal structure starting at the take-off centre.
- The *idea* of iterative refinement away from centre-targeting mirrors the iterative nature of the reference algorithm.

**Incorrect, missing, or unverified (highest priority first):**
1. **Not the mandated PathFinder/PCP algorithm.** pyxctsk uses a discretized DP+beam-search heuristic instead of the Ding et al. point-circle-point reflection/crossing method. Beam search is an incomplete search and can prune the optimal path; the reference method converges to the true optimum. Expected symptom: optimized distances slightly *longer* than FS/XCTrack for tasks with large or awkwardly placed cylinders.
2. **No LTM Cartesian projection (spec §7.1.2) and no ProjectionCorrection (§7.1.7).** The `optimal_point` geometry is computed directly in lat/lon; if it treats degrees as planar when placing the boundary point (project from centre toward target at radius r), it introduces distortion because one degree of longitude ≠ one degree of latitude in metres away from the equator. *This specific math could not be verified from the published source and must be confirmed in `turnpoint.py`.*
3. **"Crossing" case likely unhandled.** When `prev_point` (or the look-ahead target) lies *inside* the current cylinder — which happens with nested/concentric/overlapping cylinders, a small turnpoint inside a large start cylinder, or the take-off inside the SSS — Ding et al. take the segment-circle intersection, not a centre-directed projection. [MATEC Conferences](https://www.matec-conferences.org/articles/matecconf/pdf/2018/91/matecconf_eitce2018_03027.pdf) [matec-conferences](https://www.matec-conferences.org/articles/matecconf/pdf/2018/91/matecconf_eitce2018_03027.pdf) If `optimal_point` always projects toward the target at radius r, these cases produce wrong (usually too-long) legs. Needs verification and, if absent, a fix.
4. **Convergence criterion differs from ε = 0.1 m.** pyxctsk uses a fixed iteration count and breaks on the first non-improving pass; [github](https://github.com/simonsteiner/pyxctsk/blob/main/src/pyxctsk/route_optimization.py) the spec uses a 0.1 m change threshold. A single non-improving refinement pass can terminate before true convergence.
5. **`earthModel` field ignored.** XCTrack `.xctsk` tasks carry an `earthModel` field (per the xctrack.org Competition Interfaces spec, "one of 'WGS84'(default) / 'FAI_SPHERE'", used for "distance computation for cylinder sizes"); pyxctsk always uses WGS84 geodesic. For current FAI Cat 1 comps WGS84 is correct, but this will mismatch XCTrack when a task specifies the FAI sphere. Per the xctrack.org Competition Tutorial, "On 50km cylinder radius the difference between FAI sphere formula and WGS-84 formula can be about 200-300m, which can very easily result in making the cylinder in XCTrack yet missing it in the view of the scoring software (!)."
6. **Speed-section / ESS and SSS pre-start subtraction unverified.** The spec computes speed-section distance as optimized launch→ESS minus the pre-start portion (subtracting the start cylinder radius). Whether `task_distances.py`/`sss_calculations.py` implement this correctly could not be confirmed and should be tested.
7. **Goal-line orientation.** The 2026 rule orients the goal line perpendicular to the line from the *previous optimized route point* to the goal centre. Because pyxctsk resolves the goal line inside `optimal_point` using the incoming `prev_point`, this is plausibly correct, but should be tested against a known goal-line task.

## Details

The FAI 2026 RouteOptimizer pipeline is: `GeodesicToCartesian` (LTM) → `PathFinder` (Ding et al., ε = 0.1 m) → `CartesianToGeodesic` → `ProjectionCorrection` → sum of `EllipsoidDistance` over consecutive WGS84 path points. pyxctsk collapses this to: pick a discrete boundary point per turnpoint via `optimal_point`, run DP+beam search over those points measuring legs with `geopy` geodesic distance, and refine a few times. The distance metric is right; the point-selection geometry and the search strategy are approximations.

The Ding et al. paper's core primitives that pyxctsk should mirror: `GetoptPi(p_{i-1}, p_{i+1}, O_i)` returns the optimal point on circle i — using PCP if the previous point is outside the circle and the next point is in the "reflection region," otherwise the first intersection of segment p_{i-1}p_{i+1} with the circle (the crossing case). [MATEC Conferences](https://www.matec-conferences.org/articles/matecconf/pdf/2018/91/matecconf_eitce2018_03027.pdf) Algorithm 2 alternates fixing odd/even points [MATEC Conferences](https://www.matec-conferences.org/articles/matecconf/pdf/2018/91/matecconf_eitce2018_03027.pdf) and repeats until |L_new − L_old| < ε. [matec-conferences](https://www.matec-conferences.org/articles/matecconf/pdf/2018/91/matecconf_eitce2018_03027.pdf) This is exactly the structure a spec-faithful reimplementation should adopt, computed in the LTM plane.

Reference cross-checks: igc-xc-score (mmomtchev) confirms the FAI approach — its README states "The FAI recommended method for computing distance is distance on the surface of a WGS84 ellipsoid… Geographical distances are calculated on a WGS84 ellipsoid (oblate spheroid) according to the FAI's recommendations and FFVL's rules," with its high-precision mode using Vincenty "hard-coded at 60cm over the WGS84 reference ellipsoid." XCTrack supports both FAI sphere and WGS84 and exposes `earthModel` in the task format; FS (FScomp) is the CIVL reference scorer. Validating pyxctsk's optimized distances against XCTrack's displayed optimized distance and FS output on identical `.xctsk` files is the definitive acceptance test.

## Recommendations
1. **Immediate:** document that pyxctsk's optimized distance is an approximation, not a spec-faithful implementation of FAI S7F 2026 §7. Add regression tests comparing `optimized_distance` to XCTrack's optimized distance for a battery of real `.xctsk` files; flag deviations > 0.1% or > 50 m.
2. **Fix ordering:** (a) confirm/repair `optimal_point` to handle the crossing case (prev/target inside cylinder) and to place the boundary point using geodesic bearing math, not planar degrees; (b) honor the `earthModel` field; (c) replace the beam-search heuristic with the Ding et al. PCP reflection/crossing iteration in an LTM plane, converging at ε = 0.1 m; (d) verify ESS/SSS pre-start subtraction and goal-line orientation.
3. **Benchmarks that change the recommendation:** if regression tests already match XCTrack/FS within 0.1%/50 m across nested-cylinder, large-start-cylinder, and goal-line tasks, then items 1 and 3(c) can be downgraded to "documentation only" and the current heuristic kept.

## Caveats
- The exact body of `turnpoint.py` (`optimal_point`, `TaskTurnpoint`, `TurnpointGeometry`), `optimization_config.py` (numeric defaults for `DEFAULT_ANGLE_STEP`, `DEFAULT_BEAM_WIDTH`, `DEFAULT_NUM_ITERATIONS`), [github](https://github.com/simonsteiner/pyxctsk/blob/main/src/pyxctsk/distance.py) `task_distances.py`, and `sss_calculations.py` could not be retrieved from the public web during this audit (GitHub raw/tree endpoints were robots-blocked; the PyPI source tarball `pyxctsk-0.4.1.tar.gz` is the reliable way to obtain them). Findings 2, 3, 6, and 7 are therefore reasoned inferences from the confirmed call sites and docstrings and must be verified against the actual source before fixing.
- pyxctsk is derived from Tom Payne's go-xctrack [github](https://github.com/simonsteiner/pyxctsk) and targets XCTrack compatibility, so some behavior is intended to match XCTrack rather than FS/FAI exactly.

## Ready-to-paste Claude Code prompt

> **Context:** `pyxctsk` (github.com/simonsteiner/pyxctsk) is a Python implementation of XCTrack's `.xctsk` task format. Its route/distance optimization lives in `src/pyxctsk/route_optimization.py` (DP + beam search + iterative refinement), `src/pyxctsk/turnpoint.py` (`TaskTurnpoint`/`TurnpointGeometry.optimal_point`, `distance_through_centers`), `src/pyxctsk/optimization_config.py`, `src/pyxctsk/task_distances.py`, and `src/pyxctsk/sss_calculations.py`. Leg distances use `geopy.distance.geodesic` (WGS84/Karney).
>
> **Goal:** make the optimized task-distance calculation faithful to FAI Sporting Code S7F 2026 §7 (task distance = shortest path launch→goal touching each cylinder/line in order) and to the algorithm it cites, Ding, Xie & Jiang, "An Efficient Algorithm for Touring n Circles" (MATEC Web of Conferences 232, 03027, EITCE 2018, DOI 10.1051/matecconf/201823203027).
>
> **Do the following:**
> 1. **Audit `turnpoint.py` `optimal_point(prev_point, next_target)`** and report exactly how it places the point on a cylinder. If it projects from the centre toward the target using raw lat/lon differences (treating degrees as planar), change it to use geodesic bearing/destination math (`geopy.distance.geodesic(...).destination` or geographiclib `Geodesic.WGS84.Direct`) so the point sits at radius `r` on the true ellipsoidal boundary.
> 2. **Add the "crossing" case:** if `prev_point` or `next_target` lies *inside* the cylinder (distance to centre < r), the optimal point must be the intersection of the segment prev→target with the circle boundary, per Ding et al., not a centre-directed projection. Cover nested/concentric/overlapping cylinders and the take-off-inside-SSS case.
> 3. **Replace the beam-search heuristic** in `route_optimization.py` with the Ding et al. iterative point-circle-point method: initialize a point per turnpoint, then alternately fix odd/even points and update each via a `GetoptPi(prev, next, cylinder)` that chooses reflection (PCP) vs. crossing; iterate until the change in total path length < **ε = 0.1 m** (spec §7.1.3). Perform the point-placement in a localized Transverse Mercator plane centred on the task-area centre (spec §7.1.2), then convert back to WGS84 and snap each point to its boundary (ProjectionCorrection, §7.1.7) before summing `geodesic` leg distances.
> 4. **Honor the `earthModel` field** from the `.xctsk` task ("WGS84" default vs "FAI_SPHERE", radius 6371 km): when FAI sphere is specified, compute leg distances on the sphere (great-circle, R = 6 371 000 m) and place boundary points accordingly; otherwise keep WGS84 geodesic.
> 5. **Verify ESS/SSS handling** in `task_distances.py`/`sss_calculations.py`: speed-section distance = optimized launch→ESS distance minus the pre-start portion (subtract the SSS cylinder radius as in spec §7.2). Verify the goal line is oriented perpendicular to the line from the previous optimized route point to the goal centre (§6.2.3.1).
>
> **Acceptance criteria / tests (add to `tests/test_distance.py`):**
> - For 5+ real `.xctsk` files, `optimized_distance` matches XCTrack's displayed optimized distance within **0.1% and 50 m**.
> - Nested cylinders (a small turnpoint fully inside a large start cylinder) return a path no longer than the straight launch→…→goal geodesic through boundary points, and never route "out and back" spuriously.
> - A task where the previous point is inside the current cylinder yields the segment-circle intersection (crossing case), verified against a hand-computed value.
> - A goal-line task: the finish point lies on the goal line perpendicular to the incoming optimized leg; distance matches XCTrack.
> - Convergence: increasing `num_iterations` beyond the point where the total length changes by < 0.1 m does not change the result by > 0.1 m.
> - Regression: existing tests still pass; `optimized_distance` is always ≤ `distance_through_centers`.
